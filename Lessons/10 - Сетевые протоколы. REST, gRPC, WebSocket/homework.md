# ДЗ к занятию 10. Сколько стоит JSON? Расчёт экономии gRPC vs REST/JSON

## Цель

Научиться **считать**, а не «чувствовать» выбор протокола. Перевести абстрактное «gRPC компактнее» в конкретные цифры: гигабайты трафика, доллары, миллисекунды и ядра CPU. Это тот же навык cost estimation, что и в расчётах сайзинга и build-vs-buy — только применённый к протоколу межсервисного взаимодействия.

**Главный тезис:** «компактнее в 5 раз» — это не аргумент. Аргумент — это «экономит $X в год при нашей нагрузке, и вот когда это окупает переписывание».

---

## Контекст задачи

Вы — архитектор в системе из микросервисов. Два внутренних сервиса (`feed-service` → `recommendation-service`) общаются по REST/JSON. Поступило предложение перевести их на gRPC. Нужно обосновать решение цифрами, а не «потому что модно».

---

## Часть 1. Базовый расчёт (по заданным параметрам)

### Исходные данные

| Параметр | Значение |
|----------|----------|
| RPS между двумя сервисами (пик) | 50 000 req/s |
| Средний RPS (с учётом суток) | 20 000 req/s |
| Размер ответа в JSON (несжатый) | 4 КБ |
| Коэффициент компактности Protobuf vs JSON | в 4 раза меньше payload |
| Эффект gzip на JSON | сжимает в ~3 раза |
| Цена egress / межзонального трафика | 1 ₽ за ГБ |
| HTTP/1.1 (REST): overhead заголовков на запрос | ~0.8 КБ |
| HTTP/2 (gRPC): overhead заголовков после HPACK | ~0.05 КБ |

### Что посчитать

1. **Объём трафика в сутки** для каждого варианта (учитывайте средний RPS):
   - REST/JSON без сжатия
   - REST/JSON + gzip
   - gRPC/Protobuf (payload в 4× меньше + HTTP/2 заголовки)
2. **Объём трафика в месяц** (30 дней) для каждого варианта.
3. **Экономия в ГБ и в $/месяц и $/год** при переходе gRPC vs (REST+gzip).
4. **Пиковая полоса (Гбит/с)** при 50 000 RPS для REST/JSON и для gRPC — упрётесь ли в сетевые лимиты инстанса (например, 10 Гбит/с)?

> Подсказка по методике (как в расчётах сайзинга):
> `трафик/сутки = RPS_средний × размер_сообщения × 86 400 секунд`
> Не забудьте: «размер сообщения» = payload + overhead заголовков.

## Часть 2. Скрытая стоимость — не только трафик

Трафик — самая видимая, но не единственная статья. Оцените (можно прикидочно, с обоснованием порядка величины):

1. **CPU на сериализацию.** JSON-сериализация/парсинг дороже бинарной. Прикиньте: если парсинг JSON занимает ~X мкс на сообщение, а Protobuf ~X/3, сколько ядер CPU экономится на 50 000 RPS? (Возьмите разумную оценку X и обоснуйте.)
2. **Latency.** На что влияет переход: меньше байт по сети + быстрее (де)сериализация. Грубо оцените выигрыш p50 latency на одном вызове.
3. **Стоимость перехода.** Что в минусе: переписывание контрактов в `.proto`, кодогенерация в CI, обучение команды, потеря удобной отладки (curl → grpcurl), нужен grpc-web прокси если кто-то ходит из браузера. Оцените трудозатраты в человеко-днях.

## Часть 3. Точка окупаемости (главный вывод)

Сведите экономию и стоимость перехода:

1. Посчитайте **экономию в ₽/год** (трафик + CPU, если оценили в деньгах через стоимость инстансов).
2. Оцените **разовую стоимость миграции** в ₽ (человеко-дни × стоимость дня инженера, ориентир senior ≈ 25 000 ₽/раб. день).
3. Ответьте: **за сколько месяцев окупится переход?**
4. Сделайте вывод по правилу из занятия 02 (build vs buy): **при какой нагрузке (RPS) переход на gRPC оправдан, а при какой — преждевременная оптимизация?** Найдите свой порог RPS, ниже которого экономия не стоит усилий.

## Часть 4. Где НЕ применять (проверка понимания)

Кратко (по 1-2 предложения) ответьте, опираясь на расчёт:

1. Почему этот же расчёт **не** обосновывает перевод **публичного REST API наружу** на gRPC, даже если трафика там больше? (Подумайте про кеширование на CDN — оно меняет всю экономику.)
2. В каком случае экономия трафика gRPC **мнимая** — когда JSON+gzip почти не уступает? (Подумайте про размер и характер payload.)

## Критерии оценки

1. Корректно посчитан трафик в сутки/месяц для всех трёх вариантов (JSON, JSON+gzip, gRPC).
2. Экономия выражена в ГБ и в деньгах (₽/год), не только «в разах».
3. Учтена стоимость перехода — вывод сбалансирован (экономия vs затраты), а не «gRPC всегда лучше».
4. Сформулирован **порог RPS/нагрузки**, при котором переход оправдан.
5. Часть 4: понимание, что для публичного API расчёт меняется из-за кеширования.

**Статус «Принято»** при выполнении критериев 1-4.

## Пример оформления расчёта (образец на части данных)

> Это образец хода рассуждения для REST/JSON без сжатия — остальные варианты считаете аналогично.

```
Размер сообщения (REST/JSON):  4 КБ payload + 0.8 КБ заголовки = 4.8 КБ
Трафик в сутки = 20 000 RPS × 4.8 КБ × 86 400 с
              = 20 000 × 4.8 × 86 400 КБ
              ≈ 8 294 400 000 КБ ≈ 7.7 ТБ/сутки
Трафик в месяц ≈ 7.7 ТБ × 30 ≈ 231 ТБ/мес
Стоимость = 231 000 ГБ × 1 ₽ ≈ 231 000 ₽/мес ≈ 2,8 млн ₽/год
```

> Дальше: посчитайте то же для JSON+gzip и для gRPC, найдите разницу — это и есть экономия. Сравните с разовой стоимостью миграции и сделайте вывод.
