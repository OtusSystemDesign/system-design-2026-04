# ADR-0002. Транскодинг через очередь заданий (асинхронный медиа-пайплайн)

- **Status:** Accepted
- **Date:** 2026-04
- **Context.** Транскодинг 2-часового тайтла в полную ABR-лесенку — это десятки часов CPU/GPU, выполняемые параллельно по сегментам; поток поступления неравномерен (~200–500 тайтлов/неделю всплесками от студий). Делать это синхронно в момент загрузки нельзя: процесс долгий, ресурсоёмкий и подвержен сбоям. Рассматривались: синхронная обработка на ingest, ручной запуск, асинхронная очередь заданий с пулом воркеров.
- **Decision.** Ingest лишь принимает мастер-файл и кладёт задание в очередь (Kafka/SQS); пул transcoding-воркеров забирает задания, проходит конечный автомат `queued → transcoding → packaging → published` (см. uml.md) и публикует результат в storage и каталог.
- **Consequences.**
  - **+** Всплески поглощаются буфером очереди; ферма воркеров масштабируется независимо (в т.ч. на spot/GPU-инстансах).
  - **+** Сбой одного задания изолирован: ретраи и dead-letter не блокируют остальной поток.
  - **+** Path воспроизведения полностью отделён от тяжёлого пайплайна — нагрузка на read path не зависит от публикаций.
  - **−** Eventual consistency: тайтл доступен не сразу после загрузки, а после прохождения пайплайна → нужна явная модель статусов для контент-менеджеров.
  - **−** Растёт операционная сложность: идемпотентность воркеров по jobId, мониторинг лага очереди, обработка частичных сбоев и dead-letter.
  - **−** Сквозную отладку «загрузка → публикация» труднее проследить — обязательна распределённая трассировка и корреляция по jobId.
