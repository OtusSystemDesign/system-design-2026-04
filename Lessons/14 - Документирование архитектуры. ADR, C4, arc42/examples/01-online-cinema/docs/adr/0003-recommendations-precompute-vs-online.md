# ADR-0003. Рекомендации: precompute (батч) как основа + лёгкий online-реранкинг

- **Status:** Accepted
- **Date:** 2026-04
- **Context.** Главный экран должен открываться с персональными подборками для ~10 млн пользователей за p95 < 300 мс, при этом модели рекомендаций тяжёлые (коллаборативная фильтрация по матрице просмотров). Считать полную выдачу синхронно в запросе невозможно по latency и стоимости. Рассматривались: полностью online inference на каждый запрос, чистый precompute (ночной батч), гибрид.
- **Decision.** Тяжёлые подборки считаем батчем offline и кладём готовыми в Reco Store (key-value по userId); в запросе делаем только лёгкий online-реранкинг (свежие события сессии, «продолжить просмотр», бизнес-правила и дедуп).
- **Consequences.**
  - **+** Чтение из key-value по ключу — стабильные единицы миллисекунд, легко выдержать пик главного экрана.
  - **+** Тяжёлые вычисления вынесены в батч → предсказуемая нагрузка и стоимость, проще масштабировать independently.
  - **+** Online-реранкинг даёт реакцию на текущую сессию без полного пересчёта модели.
  - **−** Свежесть ограничена циклом батча: новый интерес пользователя попадёт в основную выдачу не сразу (минимизируется реранкингом, но не устраняется).
  - **−** Двойная система (батч-пайплайн + online-слой) — больше кода, мониторинга и точек отказа, чем у одного подхода.
  - **−** «Холодный старт» новых пользователей/тайтлов требует отдельной fallback-логики (популярное/трендовое).
