# ADR-0004. Хранилище метаданных каталога — PostgreSQL

- **Status:** Accepted
- **Date:** 2026-04
- **Context.** Каталог — ~30 000 тайтлов с метаданными (название, жанры, актёры, обложки, рендишены, права и доступность по регионам). Данные требуют целостности (тайтл ↔ рендишены ↔ региональные права), фильтрации по многим атрибутам и согласованной публикации; объём небольшой и помещается в одну управляемую БД, но read-нагрузка высокая. Рассматривались: документная БД (MongoDB) ради гибкой схемы, PostgreSQL с реляционной моделью + кэш.
- **Decision.** Метаданные каталога храним в PostgreSQL (реляционная модель с FK и транзакциями), горячее чтение закрываем кэшем (Redis) и read-репликами; поиск при необходимости выносим в отдельный индекс (Elasticsearch).
- **Consequences.**
  - **+** Транзакции и внешние ключи гарантируют целостность связей тайтл/рендишены/права при публикации.
  - **+** Гибкие выборки и джойны по атрибутам без денормализации; знакомый командам инструмент, зрелая эксплуатация.
  - **+** Объём каталога невелик → одна primary-инстанс + реплики легко обслуживают нагрузку чтения.
  - **−** Жёсткая схема: добавление нетипичных атрибутов под отдельные типы контента требует миграций.
  - **−** Масштабирование записи ограничено одним primary (для каталога не проблема, но это явный потолок).
  - **−** Полнотекстовый/фасетный поиск в Postgres ограничен → при росте требований появляется отдельный поисковый индекс и задача его синхронизации.

## Рассмотренные альтернативы

> Образец фиксации того, *почему* вариант отклонён. **Дисквалифицирующий критерий** — обязательное требование, которое вариант не выполняет; такой вариант отклоняется, даже если по другим осям он выигрывает.

### Вариант B — документная БД (MongoDB)
- ➕ Гибкая схема: разнотипные атрибуты под разные виды контента без миграций.
- ➕ Простое горизонтальное масштабирование записи.
- ⛔ **Дисквалифицирующий критерий (целостность):** публикация тайтла атомарно меняет 3 связанные сущности (тайтл ↔ рендишены ↔ региональные права). Это hard requirement; eventual consistency документной модели допускает окно, в котором тайтл виден без прав/рендишенов → **вариант отклонён**.

### Вариант C — только Elasticsearch как основное хранилище
- ➕ Из коробки закрывает фасетный/полнотекстовый поиск (одна из задач каталога).
- ⛔ **Дисквалифицирующий критерий (source of truth):** Elasticsearch — производный индекс, не транзакционный store; делать его системой записи означает терять гарантии целостности и усложнять восстановление. Используем его **в дополнение** к PostgreSQL (см. минусы выше), но не как primary → **как основное хранилище отклонён**.

**Применимость решения (границы).** PostgreSQL выбран для **метаданных каталога** (небольшой объём, высокая целостность). Он **не предназначен** для горячих координат/телеметрии воспроизведения и для медиа-сегментов — там действуют отдельные решения (Redis, object storage). См. §11 arc42.
