Минимальный связный каркас по шаблону arc42. Документ — точка входа: §5 ссылается на [`c4.md`](c4.md), §6 — на [`uml.md`](uml.md), §9 — на [`../adr/`](../adr/).

## §1 Introduction and Goals

Онлайн-кинотеатр — подписочный видеостриминг (SVOD): каталог, адаптивное воспроизведение (ABR) через CDN, асинхронный медиа-пайплайн транскодинга и персональные рекомендации. Система read-heavy (read/write ≈ 1000:1): ~10 млн подписчиков, до 3–4 млн одновременных потоков на вечернем пике. Подробный контекст — в [`../../README.md`](../../README.md).

**Quality goals (в цифрах):**

| Цель | Метрика |
|------|---------|
| Быстрый старт воспроизведения | Время от Play до первого кадра p95 < 2 с |
| Плавность просмотра | Доля времени в ребуферинге < 0.5% (rebuffer ratio) |
| Доступность в прайм-тайм | Доступность пути воспроизведения 99.95% |
| Отзывчивость главного экрана | Выдача рекомендаций p95 < 300 мс |
| Эффективность раздачи | CDN cache hit ratio > 95% (origin offload) |

**Стейкхолдеры:** зритель, контент-менеджер, команда эксплуатации (SRE), правообладатели/студии (DRM-требования), биллинг.

---

## §2 Constraints

- **Защита контента:** DRM (Widevine/FairPlay) обязателен по договорам со студиями; ключи не покидают DRM-провайдера.
- **Регуляторика/лицензии:** доступность тайтлов ограничена по регионам (geo-fencing).
- **Совместимость:** парк устройств от мобильных до Smart TV → нужны HLS и DASH.
- **Стоимость egress:** видео-трафик нельзя гнать через прикладные сервисы — только CDN.
- **Команды и стек:** Go/Java/Python, Kubernetes, S3-совместимое storage, managed Kafka/Redis/PostgreSQL.

## §4 Solution Strategy

| Решение | Подход | ADR |
|---------|--------|-----|
| Доставка видео | ABR (HLS/DASH) + раздача с CDN edge, origin offload | [ADR-0001](../adr/0001-abr-streaming-hls-dash-cdn.md) |
| Подготовка контента | Асинхронный медиа-пайплайн на очереди заданий | [ADR-0002](../adr/0002-transcoding-job-queue.md) |
| Рекомендации | Precompute (батч) + лёгкий online-реранкинг | [ADR-0003](../adr/0003-recommendations-precompute-vs-online.md) |
| Метаданные каталога | PostgreSQL + кэш/реплики, поиск отдельным индексом | [ADR-0004](../adr/0004-catalog-metadata-postgresql.md) |

Сквозной принцип: разделение read path (быстрый, кэшируемый, через CDN) и write/pipeline path (тяжёлый, асинхронный) — они масштабируются и отказывают независимо.

## §5 Building Block View

Структура системы описана C4-диаграммами — см. [`c4.md`](c4.md):
- **Level 1 — System Context:** граница системы, акторы, внешние системы (CDN, DRM, payment, IdP).
- **Level 2 — Container:** прикладные сервисы, хранилища, очередь и медиа-пайплайн.
- **Level 3 — Component:** разложение Playback Service (entitlements, manifest builder, URL signer, DRM proxy).

---

## §6 Runtime View

Ключевые сценарии — см. [`uml.md`](uml.md):
- **Старт воспроизведения через CDN (sequence).** Плеер получает манифест и подписанные URL от Playback Service, берёт DRM-лицензию, качает сегменты с CDN edge (с тягой из origin при промахе) и адаптирует битрейт по скорости сети. Показывает соблюдение quality goal «p95 < 2 с» и offload трафика на CDN.
- **Жизненный цикл задания транскодинга (state machine).** Задание проходит `uploaded → queued → transcoding → packaging → published` с ветками `failed → retry → dead-letter`. Контракт для мониторинга лага очереди и идемпотентности воркеров.

Физическое размещение по зонам (edge → origin → core → pipeline) — deployment-диаграмма там же.

## §8 Crosscutting Concepts

- **Безопасность контента.** DRM-шифрование сегментов; Playback Service выдаёт только подписанные временные URL (без прямого доступа к storage); проверка entitlements (подписка, регион, родительский контроль) до выдачи манифеста.
- **Наблюдаемость.** Клиентская QoE-телеметрия (битрейт, rebuffer ratio, время старта) как первичный SLI; серверные метрики latency/availability пути воспроизведения; мониторинг лага очереди транскодинга и dead-letter. Распределённая трассировка по jobId/sessionId.

- [ADR-0003](../adr/0003-recommendations-precompute-vs-online.md) — Рекомендации: precompute + online-реранкинг (Accepted)

ADR-0004 дополнительно показывает фиксацию отвергнутых *вариантов внутри* принятого решения (секция «Рассмотренные альтернативы» с дисквалифицирующим критерием).

## §11 Risks and Technical Debt

- **Зависимость от CDN.** Деградация/смена CDN-вендора напрямую бьёт по quality goals; митигация — мульти-CDN и origin shield, но это усложняет инвалидацию и прогрев.
- **Стоимость и время транскодинга.** Полная ABR-лесенка × 2 формата для 4K дорога по storage и CPU/GPU; риск роста backlog при всплесках публикаций — нужен контроль лага очереди и приоритизация заданий.
- **Свежесть рекомендаций.** Батч-цикл ограничивает реакцию на новый интерес; при недостаточном online-реранкинге страдает релевантность и удержание.
