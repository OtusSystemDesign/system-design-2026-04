# ADR-0001. База данных на сервис (database per service) вместо общей монолитной БД

- **Status:** Accepted
- **Date:** 2026-04
- **Context.** Домены маркетплейса (каталог, заказы, оплата, сток, доставка) имеют разный профиль нагрузки и темп изменений: каталог — read-heavy и миллионы SKU, заказы — write-heavy с пиками распродаж, оплата — строгие требования к консистентности и аудиту. Общая монолитная БД создаёт связанность схем, делает независимый деплой и масштабирование невозможными, а блокировки одного домена влияют на другие. Рассматривались: единая БД с разделением по схемам vs отдельная БД на каждый сервис.
- **Decision.** Каждый микросервис владеет собственной БД, недоступной другим сервисам напрямую; обмен данными — только через API и события (Kafka).
- **Consequences.**
  - **+** Независимый деплой, масштабирование и выбор хранилища под нагрузку домена (PostgreSQL для заказов, OpenSearch для поиска, Redis для корзин).
  - **+** Изоляция отказов и нагрузки: тяжёлый запрос к каталогу не блокирует оформление заказов.
  - **−** Нет распределённых ACID-транзакций между сервисами — согласованность только eventual, бизнес-операции через saga (см. [ADR-0002](0002-checkout-saga-orchestration.md)).
  - **−** Дублирование/синхронизация данных (например, проекция каталога в поисковый индекс) и рост операционной сложности: больше БД для бэкапов, миграций и мониторинга.
