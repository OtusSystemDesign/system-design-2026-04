# ADR-0003. Резервирование стока через резерв с TTL (а не только оптимистичная блокировка)

- **Status:** Accepted
- **Date:** 2026-04
- **Context.** Главное требование бизнеса — **не продать больше остатка** (oversell), особенно в пики распродаж, когда десятки покупателей конкурируют за один SKU. Между шагом резерва и завершением оплаты проходит время; если просто уменьшать остаток оптимистичной блокировкой по версии строки, при брошенной оплате товар «зависнет» списанным, а при высокой конкуренции пойдёт лавина конфликтов и повторов. Рассматривались: оптимистичная блокировка (version/CAS на остатке) vs явный резерв с TTL.
- **Decision.** На шаге saga создаётся явная запись резерва (orderId, SKU, qty) с TTL; доступный остаток = физический − активные резервы; при оплате резерв конвертируется в списание, при сбое/таймауте — освобождается (release или авто-истечение по TTL).
- **Consequences.**
  - **+** Oversell исключён: резерв уменьшает доступный остаток сразу, а TTL гарантирует возврат стока при брошенных корзинах и зависших saga без ручного вмешательства.
  - **+** Чёткая семантика для saga: компенсация — это просто release резерва; видно, сколько товара «в процессе покупки».
  - **−** Сложнее модель данных и фоновый процесс: нужна таблица резервов, джоба истечения TTL и аккуратный подсчёт доступного остатка под конкурентной записью.
  - **−** Подбор TTL — компромисс: слишком короткий рвёт медленные оплаты, слишком длинный временно «прячет» реально доступный товар в пик распродажи.
