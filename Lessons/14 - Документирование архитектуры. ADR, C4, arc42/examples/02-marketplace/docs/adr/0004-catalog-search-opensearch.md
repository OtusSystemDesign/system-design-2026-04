# ADR-0004. Поиск по каталогу на OpenSearch (а не SQL-поиск по основной БД)

- **Status:** Accepted
- **Date:** 2026-04
- **Context.** Каталог — миллионы SKU, поиск даёт десятки тысяч RPS на пике при соотношении чтение:запись ≈ 100:1. Нужны полнотекстовый поиск с морфологией, фасетная фильтрация (бренд, цена, атрибуты), ранжирование по релевантности и p95 < 300 мс. SQL `LIKE`/полнотекстовый поиск PostgreSQL не даёт нужного качества релевантности и фасетов на таком объёме, а тяжёлые поисковые запросы конкурируют за ресурсы транзакционной БД каталога. Рассматривались: поиск средствами PostgreSQL (tsvector/GIN) vs выделенный поисковый движок OpenSearch.
- **Decision.** Поиск и фасеты вынесены в OpenSearch; индекс наполняется из Catalog DB асинхронно через события изменений каталога (CDC/Kafka), а транзакционным источником истины остаётся PostgreSQL.
- **Consequences.**
  - **+** Релевантность, морфология и фасеты «из коробки», горизонтальное масштабирование чтения; поисковая нагрузка изолирована от транзакционной БД каталога.
  - **+** Профиль read-heavy масштабируется независимо — каталог остаётся доступным (99.95%) даже под пиковым поиском.
  - **−** Eventual consistency индекса: между изменением товара и его появлением в поиске есть лаг (секунды), требуется мониторинг отставания индексации.
  - **−** Отдельная инфраструктура и конвейер индексации (CDC, маппинги, переиндексация) — рост сложности эксплуатации и стоимости; дублирование данных каталога в индексе.

## Рассмотренные альтернативы

> Образец фиксации того, *почему* вариант отклонён. **Дисквалифицирующий критерий** — обязательное требование, которое вариант не выполняет; такой вариант отклоняется, даже если по другим осям он выигрывает.

### Вариант B — поиск средствами основной БД (PostgreSQL `LIKE`/полнотекст `tsvector`/GIN)
- ➕ Нет отдельной инфраструктуры и конвейера индексации: один источник данных, нет лага и задачи синхронизации.
- ➕ Сильная консистентность: товар виден в поиске сразу после коммита, без eventual-окна.
- ⛔ **Дисквалифицирующий критерий (поиск p95 < 300 мс под нагрузкой):** на миллионах SKU и десятках тысяч RPS полнотекст/фасеты в Postgres не держат latency, а тяжёлые поисковые запросы конкурируют за ресурсы транзакционной БД каталога и роняют её доступность (quality goal 99.95%). Поисковый трафик нельзя класть на primary каталога → **вариант отклонён**.

### Вариант C — OpenSearch как источник истины каталога (без PostgreSQL)
- ➕ Из коробки закрывает релевантность, морфологию и фасеты — одна система вместо двух, нет дублирования данных.
- ➕ Горизонтальное масштабирование чтения «бесплатно».
- ⛔ **Дисквалифицирующий критерий (source of truth / консистентность):** OpenSearch — производный индекс с near-real-time видимостью и без транзакций; делать его системой записи означает терять целостность каталога (цена, остаток-привязка, статус товара) и усложнять восстановление после сбоя. Используем его **в дополнение** к PostgreSQL (см. минусы выше), но не как primary → **как источник истины отклонён**.

**Применимость решения (границы).** OpenSearch выбран для **поиска и фасетов по каталогу** (read-heavy, требование релевантности и latency). Он **не предназначен** быть транзакционным хранилищем каталога, корзины, заказов или стока — там действуют отдельные решения (PostgreSQL на сервис, резерв стока с TTL). См. §4 и §9 arc42.
