# arc42 — Маркетплейс

Минимальный связный каркас arc42. Building Block View ссылается на [`c4.md`](c4.md), Runtime View — на [`uml.md`](uml.md), Architecture Decisions — на [`../adr/`](../adr/). Это не четыре отдельных документа, а единое описание.

## §1 Introduction and Goals

Маркетплейс связывает покупателей и продавцов: каталог и поиск товаров, корзина, оформление заказа (checkout-saga), оплата, резервирование стока и доставка. Масштаб — ~5M активных покупателей, сотни тысяч продавцов, миллионы SKU, ~500K заказов в сутки с пиками до 3–5× на распродажах.

**Quality goals (в цифрах):**

| Цель | Метрика |
|------|---------|
| Быстрое оформление заказа | checkout p95 < 1.5 с |
| Консистентность стока | oversell = 0 (не продать больше остатка) |
| Доступность каталога | 99.95% (каталог доступен даже при деградации checkout) |
| Быстрый поиск | поиск p95 < 300 мс |
| Согласованность денег и стока | при отмене после оплаты — refund 100%, резерв снят |

**Стейкхолдеры:** покупатель, продавец, оператор поддержки, команда платежей/биллинга, команда инфраструктуры.

## §2 Constraints

- Платёж, доставка и антифрод — внешние системы; интеграция только через адаптеры с таймаутами и идемпотентностью.
- Нет распределённых ACID-транзакций между сервисами (database per service) — согласованность eventual.
- Требования регуляторов к хранению платёжных данных и аудиту — оплата изолирована в отдельном сервисе/БД.
- Пиковые распродажи задают расчётную нагрузку и SLA на стоковую консистентность.

## §4 Solution Strategy

- **Микросервисы с БД на сервис** за единым API Gateway/BFF — изоляция нагрузки и независимый деплой (см. [ADR-0001](../adr/0001-database-per-service.md)).
- **Checkout как saga с оркестрацией** — бизнес-атомарность с компенсациями вместо распределённых транзакций (см. [ADR-0002](../adr/0002-checkout-saga-orchestration.md)).
- **Резерв стока с TTL** — гарантия отсутствия oversell и автоматический возврат брошенных резервов (см. [ADR-0003](../adr/0003-stock-reservation-ttl.md)).
- **Поиск на OpenSearch с асинхронной индексацией** — read-heavy каталог масштабируется независимо от транзакционной БД (см. [ADR-0004](../adr/0004-catalog-search-opensearch.md)).

## §6 Runtime View

Динамика описана в [`uml.md`](uml.md). Ключевые сценарии:

- **Checkout-saga (happy path и компенсации)** — sequence-диаграмма: оплата → резерв стока → отгрузка; при сбое шага оркестратор откатывает выполненные шаги в обратном порядке (release резерва, refund оплаты), заказ переходит в CANCELLED. Это прямая реализация quality goals «консистентность стока» и «согласованность денег и стока».
- **Жизненный цикл заказа** — state machine: Created → Paid → Reserved → Shipped → Delivered, с ветвями Cancelled и Refunded. Переходы строго направлены — нельзя отгрузить неоплаченный заказ.

## §8 Crosscutting Concepts

- **Надёжность и идемпотентность.** Все шаги saga и их компенсации идемпотентны (по orderId/ключу операции); события публикуются паттерном Transactional Outbox, что исключает потерю/дублирование при повторах и сбоях.
- **Наблюдаемость.** Сквозная распределённая трассировка (trace per checkout), метрики по шагам saga, лагу индексации поиска и доле успешных оплат; алерты на рост компенсаций и отставание индекса каталога.

## §11 Risks and Technical Debt

- **Order Service как точка координации.** Оркестратор saga знает обо всех участниках — риск связанности и узкого места в пик; митигация — горизонтальное масштабирование оркестратора и персистентное состояние saga.
- **Лаг индексации поиска.** Eventual consistency между каталогом и OpenSearch — товар может появляться/исчезать из поиска с задержкой; нужен мониторинг отставания и переиндексация.
- **Подбор TTL резерва.** Неверный TTL ведёт либо к срыву медленных оплат, либо к временному «скрытию» доступного товара в пик; требует калибровки по реальным данным.
