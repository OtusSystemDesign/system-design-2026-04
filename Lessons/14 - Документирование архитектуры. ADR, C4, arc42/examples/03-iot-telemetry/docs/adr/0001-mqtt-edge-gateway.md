# ADR-0001. Приём телеметрии по MQTT через edge gateway, а не прямой HTTP с устройств

- **Status:** Accepted
- **Date:** 2026-04
- **Context.** ~2M устройств шлют телеметрию раз в 10–30 с (пик ~300K точек/с) с промышленных объектов, где канал до облака нестабилен, а сами устройства маломощны и часто не имеют публичного IP. Нужен дешёвый по трафику протокол, переживающий разрывы связи без потери данных, с буфером на стороне объекта. Рассматривались: прямой HTTPS POST с каждого устройства, MQTT через edge gateway, CoAP.
- **Decision.** Устройства отдают данные локальному **edge gateway** (Modbus/OPC-UA), а gateway публикует агрегированную телеметрию в облако по **MQTT/TLS (QoS1)** с локальным store-and-forward буфером.
- **Consequences.**
  - **+** MQTT экономичен по заголовкам и держит постоянное соединение — на порядки меньше накладных расходов, чем HTTP-хендшейк на каждую точку.
  - **+** Gateway буферизует данные при разрыве канала (store-and-forward) → потеря < 0.1% даже при недоступности облака на минуты.
  - **+** Меньше точек подключения к облаку: десятки тысяч шлюзов вместо 2M устройств — проще аутентификация и rate-limiting.
  - **−** Появляется дополнительный узел (gateway) с собственным жизненным циклом: его надо разворачивать, обновлять и мониторить на объектах.
  - **−** MQTT-брокер — stateful-инфраструктура: кластеризация, sticky-сессии, persistence сложнее в эксплуатации, чем stateless HTTP за балансировщиком.
  - **−** QoS1 даёт «at least once» → возможны дубли, обработка должна быть идемпотентной (дедуп по `deviceId + timestamp`).
