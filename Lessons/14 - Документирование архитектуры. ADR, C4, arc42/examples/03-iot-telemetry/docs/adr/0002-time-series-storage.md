# ADR-0002. Хранение телеметрии в time-series БД (TimescaleDB), а не в обычной реляционной

- **Status:** Accepted
- **Date:** 2026-04
- **Context.** Поток ~100K точек/с (пик 300K), запросы — преимущественно по диапазону времени и устройству, с агрегацией (avg/min/max/rate) по окнам; объём сырья ~20–25 ТБ/год. Обычная реляционная таблица деградирует на таком темпе вставок и на time-range сканах, а ретеншн/даунсемплинг приходится писать руками. Рассматривались: PostgreSQL «как есть», TimescaleDB (расширение PG), InfluxDB.
- **Decision.** Хранить телеметрию в **TimescaleDB**: hypertable с партиционированием по времени, continuous aggregates для даунсемплинга и встроенные политики ретеншна.
- **Consequences.**
  - **+** Чанкинг по времени делает вставку и time-range запросы O(окно), а не O(вся таблица); компрессия старых чанков экономит хранилище в разы.
  - **+** Continuous aggregates и retention policy закрывают даунсемплинг и удаление сырья декларативно, без отдельных cron-джоб.
  - **+** Это PostgreSQL: знакомый SQL, JOIN с метаданными устройств, существующая экспертиза и инструменты.
  - **−** Один движок для горячего приёма и тяжёлых аналитических запросов — нужна изоляция нагрузки (реплики для чтения), иначе дашборды бьют по приёму.
  - **−** Горизонтальное масштабирование записи ограничено: за пределами одного мощного primary потребуется шардирование/распределённый режим, что усложняет эксплуатацию.
  - **−** Привязка к специфике TimescaleDB (hypertable, политики) — миграция на другой движок не бесплатна.

## Рассмотренные альтернативы

> Образец фиксации того, *почему* вариант отклонён. **Дисквалифицирующий критерий** — обязательное требование, которое вариант не выполняет; такой вариант отклоняется, даже если по другим осям он выигрывает.

### Вариант B — обычная реляционная БД (PostgreSQL «как есть») для сырой телеметрии
- ➕ Нулевая привязка к спецдвижку: знакомый SQL, любой managed-Postgres, простая эксплуатация и бэкапы.
- ➕ JOIN с метаданными устройств без расширений, единый инструментарий.
- ⛔ **Дисквалифицирующий критерий (пропускная способность записи):** quality goal — устойчивый приём **300K точек/с** на пике (§1 arc42). Плоская таблица с B-tree индексами по `(deviceId, timestamp)` на таком темпе вставок деградирует: рост и фрагментация индексов, распухание WAL, падение throughput по мере увеличения таблицы; time-range сканы идут по всей таблице, а не по чанку. Удержать 300K точек/с и при этом отдавать данные на чтение за p95 < 2 с обычная реляционка не может → **вариант отклонён**.
- ⛔ **Дисквалифицирующий критерий (стоимость хранения и ретеншн):** объём сырья ~20–25 ТБ/год; нужен ретеншн 30 дней + даунсемплинг с сохранением трендов 2 года (§1). В «голом» Postgres это руками написанные cron-джобы удаления/агрегации и отсутствие компрессии старых данных → стоимость индексов и хранилища выходит за бюджет, а ретеншн становится источником ошибок → **вариант отклонён**.

### Вариант C — хранить всё сырьё «вечно» в hot-хранилище без даунсемплинга
- ➕ Любой исторический запрос идёт по полному разрешению — не нужно думать о политиках агрегации заранее.
- ⛔ **Дисквалифицирующий критерий (стоимость/ретеншн):** при 20–25 ТБ/год полного разрешения в горячем хранилище стоимость растёт линейно и без потолка, тогда как quality goal требует контролируемой стоимости (ретеншн сырья 30 дней + даунсемплинг, тренды 2 года). Хранить всё вечно в hot-слое экономически недопустимо, а для трендов достаточно агрегатов → **вариант отклонён** в пользу continuous aggregates + retention policy.

**Применимость решения (границы).** TimescaleDB выбран для **time-series телеметрии** (высокий темп вставок по времени, диапазонные агрегаты, декларативный ретеншн). Он **не предназначен** для каталога устройств и порогов (это Device Registry на обычной реляционной модели) и для буферизации/переигрывания потока (эту роль играет Kafka — см. §4 arc42, ADR-0003). См. §11 arc42.
