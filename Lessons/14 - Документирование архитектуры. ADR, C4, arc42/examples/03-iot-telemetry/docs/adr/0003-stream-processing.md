# ADR-0003. Обработка телеметрии в потоке (Flink), а не батчами

- **Status:** Accepted
- **Date:** 2026-04
- **Context.** Бизнес требует, чтобы данные были доступны на чтение почти в реальном времени (p95 ingest→read < 2 с) и чтобы алерт срабатывал за секунды (< 5 с). Батч-обработка с интервалом в минуты этих целей в принципе не достигает. При этом обработка должна считать оконные агрегаты и оценивать правила на потоке 100–300K точек/с с гарантией exactly-once при записи. Рассматривались: микробатчи (Spark Structured Streaming с минутными интервалами), continuous stream (Flink/Kafka Streams), классический ночной батч.
- **Decision.** Обрабатывать поток **stream-движком (Apache Flink)** поверх Kafka: оконные агрегаты, обогащение и правила вычисляются по мере поступления, с checkpoint-ами и exactly-once sink в TimescaleDB.
- **Consequences.**
  - **+** Низкая end-to-end задержка: точка доступна на чтение и проверяется на алерт за секунды, а не за минуты/часы.
  - **+** Встроенные event-time окна, watermark-и и состояние операторов закрывают агрегаты и опоздавшие данные «из коробки».
  - **+** Checkpoint + exactly-once sink дают надёжность доставки в хранилище без дублей даже при падении воркеров.
  - **−** Stateful stream-инфраструктура (Flink + хранилище checkpoint-ов) сложнее в эксплуатации, чем периодический батч-джоб: бэкпрешер, рестарты, управление состоянием.
  - **−** Изменение логики/правил требует осторожного релиза с сохранением состояния (savepoints); «просто перезапустить» нельзя.
  - **−** Тяжёлые перерасчёты задним числом (backfill истории по новым правилам) в стриме неудобны — для них всё равно нужен отдельный батч-проход.
