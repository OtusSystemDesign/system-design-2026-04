# ADR-0004. Пороговые правила вычисляются в потоке, а состояние алертов — в отдельном Alert Service

- **Status:** Accepted
- **Date:** 2026-04
- **Context.** Алерт должен подниматься за < 5 с после отклонения. Правила — пороги по мгновенному значению и по оконным агрегатам (напр. «средняя температура за 60 с > 80°C»), их десятки тысяч и они меняются. Нужно решить, *где* считать условие и *где* держать состояние алерта (дедупликация, for-окно, эскалация, auto-resolve). Рассматривались: внешний движок правил, опрашивающий БД по расписанию; вычисление условий в стрим-пайплайне; полностью отдельный сервис, читающий ту же телеметрию.
- **Decision.** **Условия правил вычисляются прямо в Flink-пайплайне** (данные и так в памяти оператора), а **жизненный цикл алерта** (Pending/Firing/Escalated/Resolved, дедуп, нотификации) выносится в отдельный **Alert/Notification Service**, получающий «сырые» срабатывания через Kafka-топик.
- **Consequences.**
  - **+** Минимальная задержка: правило проверяется на той же точке, что уже обработана, без отдельного опроса БД → укладываемся в < 5 с.
  - **+** Чёткое разделение ответственности: стрим считает «условие истинно», Alert Service решает «слать ли нотификацию» (антидребезг, дедуп, эскалация) — каждую часть можно менять независимо.
  - **+** Пороги читаются из кэша метаданных в стриме, обновляются без редеплоя пайплайна.
  - **−** Логика алертинга раздвоена между двумя компонентами — нужно держать контракт alert-события и не «размазывать» правила по обоим местам.
  - **−** Сложные правила, которым нужна history/корреляция между устройствами, плохо ложатся в потоковый оператор — для них в будущем понадобится отдельный rule-движок.
  - **−** Alert Service становится stateful (состояние каждого открытого алерта) и критичным: его недоступность задерживает доставку оповещений, хотя сами срабатывания не теряются (буфер в Kafka).
