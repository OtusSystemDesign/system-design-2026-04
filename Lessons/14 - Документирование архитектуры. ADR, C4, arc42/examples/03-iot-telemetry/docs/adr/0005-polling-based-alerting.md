# ADR-0005. Алертинг через периодический опрос БД (polling) вместо потоковых правил

> 📌 **Образец отвергнутого решения.** Этот ADR показывает, как фиксируется решение, которое **рассмотрели и не приняли**. Запись не удаляют — она остаётся в журнале, чтобы к этому варианту не возвращались заново, не зная причин. Сравните со статусом `Accepted` у соседних ADR.

- **Status:** Rejected (в пользу вычисления правил в потоке — см. [ADR-0003](0003-stream-processing.md) и [ADR-0004](0004-alerting-rules.md))
- **Date:** 2026-04
- **Context.** Альтернатива вычислению пороговых правил в стрим-пайплайне (ADR-0004): не считать условия по мере поступления точки, а внешним движком правил периодически опрашивать TimescaleDB (`SELECT ... WHERE timestamp > now() - interval`), сравнивать свежие значения/агрегаты с порогами и поднимать алерты. Привлекательно тем, что правила живут вне stateful-стрима, их «просто» менять SQL-ом, а пайплайн остаётся только про запись.
- **Decision.** ~~Поднимать алерты внешним движком, который раз в N секунд опрашивает БД агрегирующими запросами и сравнивает результат с порогами.~~ **Отклонено** в пользу вычисления условий в потоке (Flink), где точка уже в памяти оператора, и выноса жизненного цикла алерта в Alert Service (ADR-0003, ADR-0004).
- **Consequences (причины отказа).**
  - ⛔ **Дисквалифицирующий критерий (алерт p95 < 5 с):** задержка алерта при polling = интервал опроса + время тяжёлого агрегирующего запроса + лаг записи в БД. Чтобы уложиться в 5 с при десятках тысяч активных правил, опрашивать пришлось бы каждые 1–2 с — это нереалистично по нагрузке на хранилище. Quality goal §1 в принципе не достигается → неприемлемо.
  - ⛔ **Дисквалифицирующий критерий (приём 300K точек/с и доступность 99.9%):** частые агрегирующие сканы по hot-данным конкурируют за тот же primary, что и приём телеметрии (минус «изоляция нагрузки» из ADR-0002). Опрос десятков тысяч правил бьёт по ingest → риск деградации приёма и падения доступности приёма ниже 99.9% (§1). Алертинг не должен ставить под угрозу приём.
  - ➖ Хуже надёжность и точность: между опросами короткие выбросы могут быть пропущены (алерт «проспал» отклонение), а агрегаты event-time с опоздавшими данными polling-ом считаются неточно — нет watermark-ов, как у стрима.
  - ➖ Дедупликация, for-окно и эскалация всё равно нужны и при polling — то есть отдельный stateful Alert Service из ADR-0004 не исчезает, а polling лишь добавляет к нему вторую нагрузку на БД.
  - ➕ (что мы потеряли, отказавшись) Простота смены правил «обычным SQL» вне стрима — это реальное удобство polling; **частично закрываем** его иначе: пороги в ADR-0004 читаются из кэша метаданных и обновляются без редеплоя пайплайна.

**Когда решение можно пересмотреть.** Если появятся правила, требующие сложной кросс-девайсной корреляции или анализа длинной истории, которые плохо ложатся в потоковый оператор (см. минусы ADR-0004), может понадобиться отдельный rule-движок, работающий по запросу — но уже над **материализованными агрегатами** (continuous aggregates), а не над сырым hot-приёмом, и только для несрочных правил, не подпадающих под порог < 5 с. Тогда этот ADR будет заменён новым (`Superseded by ADR-XXXX`), а не просто переоткрыт.
