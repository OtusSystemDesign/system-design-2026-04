# arc42 — Платформа IoT-телеметрии

> Минимальный, но связный каркас. §5 ссылается на [`c4.md`](c4.md), §6 — на [`uml.md`](uml.md), §9 — на [`../adr/`](../adr/).

## §1 Introduction and Goals

Платформа принимает поток телеметрии с ~2M IoT-устройств (через edge gateway), сохраняет его в time-series хранилище, обрабатывает на лету и поднимает алерты при выходе показателей за пороги. Основные потребители — операторы мониторинга, инженеры-аналитики и внешние системы (нотификации, тикеты, BI).

**Quality goals (в цифрах):**

| Цель | Метрика |
|------|---------|
| Низкая задержка доступности данных | ingest → доступно на чтение, p95 < 2 с |
| Быстрый алертинг | отклонение → нотификация, p95 < 5 с |
| Пропускная способность | устойчивый приём 300K точек/с на пике |
| Надёжность доставки | потеря телеметрии < 0.1% (включая разрывы канала) |
| Доступность | приём (ingest) 99.9% по месяцу |
| Стоимость хранения | ретеншн сырья 30 дней + даунсемплинг, без потери трендов 2 года |

**Стейкхолдеры:** оператор мониторинга, инженер/аналитик, эксплуатация платформы (SRE), владелец продукта, команда edge-устройств (контракт MQTT).

## §2 Constraints

- Канал edge→cloud нестабилен → обязателен буфер на стороне gateway; потеря при разрыве недопустима.
- Устройства маломощны и без публичного IP → подключение только через gateway, не напрямую.
- Поток точек неравномерен (пики ×3) → приём и обработка должны масштабироваться независимо.
- Облачное размещение в Kubernetes; используем managed-инфраструктуру где возможно.
- Телеметрия и алерты содержат данные о промышленных объектах → шифрование канала (TLS) и аутентификация шлюзов обязательны.

## §4 Solution Strategy

- **Edge-first приём по MQTT** с буферизацией на gateway — экономия трафика и устойчивость к разрывам (см. [ADR-0001](../adr/0001-mqtt-edge-gateway.md)).
- **Развязка приёма и обработки через Kafka** — лог сглаживает пики и даёт переигрывание; tracts чтения и записи разделены.
- **Stream-обработка на Flink** для секундной задержки и exactly-once записи (см. [ADR-0003](../adr/0003-stream-processing.md)).
- **Time-series хранилище (TimescaleDB)** с ретеншном и даунсемплингом (см. [ADR-0002](../adr/0002-time-series-storage.md)).
- **Алертинг: условия в потоке, состояние — в Alert Service** (см. [ADR-0004](../adr/0004-alerting-rules.md)).

## §5 Building Block View

Статическая декомпозиция описана в [`c4.md`](c4.md): System Context (Level 1) → Container (Level 2) → Component для контейнера «Ingestion/Processing Service» (Level 3). Ключевые контейнеры: MQTT Broker, Ingestion Bridge, Kafka, Ingestion/Processing Service (Flink), TimescaleDB, Alert/Notification Service, Device Registry, Query API, Web Dashboard.

## §6 Runtime View

Динамика — в [`uml.md`](uml.md). Ключевые сценарии:

1. **Ingest → обработка → алерт** (sequence): точка идёт device → gateway → MQTT → Kafka → Flink; параллельно пишется в TimescaleDB и проверяется на правила; при превышении порога рождается alert event и доставляется нотификация. Здесь видно, где достигается p95 < 2 с (read) и < 5 с (alert).
2. **Жизненный цикл алерта** (state machine): OK → Pending (for-окно отсекает дребезг) → Firing → Escalated/Acknowledged → Resolved, с дедупликацией по `(deviceId, ruleId)` в Alert Service.

## §8 Crosscutting Concepts

- **Надёжность доставки.** «At least once» на всём тракте: QoS1 от gateway, буфер store-and-forward, Kafka как durable-лог, Flink checkpoints + exactly-once sink. Обработка идемпотентна (дедуп по `deviceId + timestamp`).
- **Наблюдаемость.** Сквозные метрики: лаг Kafka-консьюмера, end-to-end latency точки (event-time → запись), backpressure Flink, доля карантина битых точек, время доставки нотификации. Распределённая трассировка от bridge до alert. Алерты на сам пайплайн (рост лага, падение throughput).
- **Безопасность.** TLS на MQTT, аутентификация шлюзов по клиентским сертификатам, изоляция read-нагрузки (реплики TSDB) от приёма.

- [ADR-0001](../adr/0001-mqtt-edge-gateway.md) — MQTT + edge gateway vs прямой HTTP (Accepted).
- [ADR-0002](../adr/0002-time-series-storage.md) — TimescaleDB vs реляционная БД (Accepted).
- [ADR-0003](../adr/0003-stream-processing.md) — stream processing (Flink) vs батч (Accepted).
- [ADR-0004](../adr/0004-alerting-rules.md) — алертинг в потоке + отдельный Alert Service (Accepted).

ADR-0002 дополнительно показывает фиксацию отвергнутых *вариантов внутри* принятого решения (секция «Рассмотренные альтернативы» с дисквалифицирующим критерием).

## §11 Risks and Technical Debt

- **TimescaleDB как одна точка масштабирования записи.** Рост сверх возможностей одного primary потребует шардирования/распределённого режима. *Митигирование:* реплики для чтения, компрессия, мониторинг роста; план перехода на распределённый режим.
- **Stateful-эксплуатация Flink.** Релизы с изменением логики требуют savepoints; ошибка управления состоянием грозит простоем приёма. *Митигирование:* runbook savepoint/restore, blue-green деплой джоба.
- **Раздвоенная логика алертинга.** Часть условий в стриме, часть жизненного цикла — в Alert Service: риск рассинхрона контракта и «сложных» правил, не ложащихся в поток. *Митигирование:* версионируемый контракт alert-события, выделение rule-движка при появлении кросс-девайсных правил.
