# Референсные примеры документации архитектуры

> 💡 **Как открыть:** просматривайте файлы в GitHub, VS Code, Obsidian или Typora — так корректно отрисуются Mermaid-диаграммы. В Блокноте они видны как текстовый код.

Три полных, самодостаточных примера документации одной системы в связке **README → C4 → UML → ADR → arc42**. Каждый оформлен как «живая документация» в структуре репозитория (`docs/architecture/`, `docs/adr/`), как это делается в реальных проектах.

Это **эталоны для подражания**, а не единственно правильные решения. Возьмите ближайший к вашему курсовому проекту пример, посмотрите, *как* раскрыта система и *какого уровня* детализация ожидается, — и проработайте то же самое для своей системы.

> ⚠️ **Это шире, чем минимально требуется.** Для самостоятельной работы достаточно минимального набора (C4 Context + Container и хотя бы один ADR — см. `../self-study.md`). Эти примеры намеренно сделаны полнее ожидаемого результата — чтобы показать уровень *хорошего* референса, к которому стоит стремиться, а не планку «иначе не зачёт».

## Что внутри каждого примера

| Артефакт | Файл | Что показывает |
|----------|------|----------------|
| Общее описание решения | `README.md` | Что за система, scope, акторы, масштаб в цифрах, навигация |
| C4 (Level 1–3) | `docs/architecture/c4.md` | Context → Container → Component |
| UML (3 диаграммы) | `docs/architecture/uml.md` | Динамика и структура поверх C4 |
| ADR (4 accepted + 1 rejected-образец) | `docs/adr/000N-*.md` | 4 принятых решения (`Accepted`) с честными плюсами и минусами + 1 отвергнутое (`Rejected`) как образец фиксации «решение не приняли» |
| arc42 (каркас) | `docs/architecture/arc42.md` | Связывает всё: §5→C4, §6→UML, §9→ADR |

## Три примера

| # | Пример | Архитектурный фокус | UML-диаграммы |
|---|--------|---------------------|----------------|
| 1 | [Онлайн-кинотеатр](01-online-cinema/README.md) | Read-heavy, CDN, асинхронный медиа-пайплайн транскодинга | Sequence, Deployment, State machine |
| 2 | [Маркетплейс](02-marketplace/README.md) | Микросервисы, database-per-service, saga, резервирование стока | Sequence, State machine, Activity |
| 3 | [IoT-телеметрия](03-iot-telemetry/README.md) | Поток событий, edge, time-series, stream processing, алертинг | Sequence, State machine, Deployment |

В сумме примеры покрывают разные архитектурные паттерны и все основные типы UML — выберите тот, что ближе к вашему домену.

## Как этим пользоваться

1. **Прочитайте один пример целиком** в порядке README → c4.md → uml.md → adr → arc42.md. Обратите внимание, как arc42 ссылается на C4, UML и ADR — это и есть «связный документ, а не четыре отдельных файла».
2. **Сверьтесь с чеклистом** из `../self-study.md` — примеры написаны так, чтобы проходить его.
3. **Повторите для своей системы:** та же структура папок, те же артефакты. C4 не смешивайте по уровням, в ADR пишите честные минусы, quality goals — только в цифрах.

## На что смотреть (признаки хорошей документации в этих примерах)

- На **C4 Level 1** нет контейнеров и БД — только система, акторы и внешние системы; видна граница системы.
- **Каждый ADR** в `Consequences` имеет и плюсы, **и** минусы/цену; `Decision` — одно предложение.
- **Quality goals** в arc42 §1 выражены в цифрах (p95, %, секунды), а не «быстро»/«надёжно».
- Нет анти-паттернов **Wall of Diagrams** (лишние дублирующие диаграммы) и **Documentation Theater** (разделы для галочки).
