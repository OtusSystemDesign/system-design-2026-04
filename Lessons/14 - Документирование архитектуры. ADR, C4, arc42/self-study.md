# Самостоятельная работа к занятию 14. Документирование архитектуры

> 💡 **Как открыть:** просматривайте этот файл в GitHub, VS Code, Obsidian или Typora — так корректно отрисуются Mermaid-диаграммы. В Блокноте они будут видны как текстовый код.

> **Это не ДЗ.** Формальное домашнее задание (концептуальная архитектура вашего курсового проекта) сдаётся отдельно и оценивается. Эта работа — тренажёр на готовом кейсе: попробовать C4, ADR и arc42 руками **до** того, как применять их к своему проекту. Не сдаётся, проверяется по чеклисту самостоятельно.

- **Время:** ~1.5–2 часа
- **Цель:** закрепить три инструмента в связке на одном сквозном кейсе
- **Инструмент на выбор:** Mermaid (проще всего), Structurizr DSL, PlantUML/C4-PlantUML, draw.io, Excalidraw

## Кейс: «Сервис доставки еды» (аналог Delivery Club / Яндекс Еда)

Краткие вводные (этого достаточно, додумывать детали — нормально):

- **Пользователи:** клиент (мобильное приложение), курьер (мобильное приложение), ресторан (web-кабинет)
- **Основные сценарии:** клиент выбирает ресторан → оформляет и **оплачивает заказ онлайн при оформлении** (оплата при получении — опциональный кейс) → ресторан готовит → курьер забирает и везёт → клиент получает заказ
- **Внешние системы:** платёжный провайдер, карты/геолокация, push-уведомления, SMS
- **Масштаб для прикидки:** ~500K заказов в день, пики в обед и вечером
- **Что важно бизнесу:** время оформления заказа, актуальный трекинг курьера на карте, не терять заказы при пиках

## Задание

### Часть 1. C4 — визуализация (40–50 мин)

1. **System Context (Level 1).** Нарисуйте сервис как один чёрный ящик. Покажите всех акторов (клиент, курьер, ресторан) и все внешние системы. Явно обозначьте **границу системы** — что наше, что чужое.
2. **Container (Level 2).** Раскройте чёрный ящик. Минимум: мобильные приложения, web-кабинет ресторана, API/gateway, сервис заказов, сервис трекинга курьеров, БД, очередь/брокер сообщений. Подпишите технологию и назначение каждого контейнера (1 строка).
3. **Component (Level 3) — по желанию.** Возьмите один контейнер (например, «Сервис заказов») и покажите его внутренние компоненты.

> Совет: делайте в Diagrams as Code (Mermaid C4 или Structurizr). Это и есть практика «живой документации».

### Часть 2. ADR — обоснование решений (30–40 мин)

Напишите **2 ADR** по формату `Status / Context / Decision / Consequences`. Возьмите два решения из списка (или свои):

- Трекинг курьера: слать координаты через **WebSocket** vs **поллинг по HTTP** vs **очередь + push**
- Хранилище заказов: **одна реляционная БД** vs **разделение на сервисы со своими БД**
- Пиковые нагрузки: оформление заказа **синхронно** vs **через очередь** (асинхронно)
- Геопоиск ресторанов: **PostGIS** vs **специализированный гео-индекс (Redis/Elasticsearch)**

**Обязательное требование:** в каждом ADR секция **Consequences** содержит и плюсы, **и минусы/цену** решения. ADR без честных минусов не засчитывается (даже самому себе).

### Часть 3. arc42 — каркас (20 мин)

Заполните **минимальный** скелет (не весь шаблон):

- **§1 Introduction and Goals** — 2–3 предложения о системе + **3 quality goals с цифрами** (например: оформление заказа < 3 с, трекинг обновляется ≤ 5 с, доступность 99.9%)
- **§4 Solution Strategy** — 3–4 ключевых решения крупными мазками
- **§5 Building Block View** — вставьте сюда C4-диаграммы из Части 1
- **§9 Architecture Decisions** — вставьте сюда ADR из Части 2

Цель — увидеть, как arc42 связывает C4 и ADR в один документ.

## Формат результата

Это **не сдаётся**, но оформите так, как будто кладёте в репозиторий проекта, — выберите один из вариантов:

- **Минимум — один markdown-файл** (`architecture.md`): C4-диаграммы, ADR и скелет arc42 в одном документе под заголовками. Подойдёт, если только начинаете.
- **Как в реальном проекте — папка** (рекомендуется):

  ```
  docs/
    architecture/
      arc42.md        # каркас arc42 (точка входа), §5 → C4, §9 → ADR
      c4.md           # C4 Level 1–3
    adr/
      0001-<slug>.md  # по одному ADR на файл
      0002-<slug>.md
  ```

  Именно так устроены [референсные примеры](examples/README.md) — посмотрите их структуру перед стартом.

> Диаграммы — в Mermaid (рендерятся в GitHub/VS Code/Obsidian/Typora). Главное — чтобы документ открывался и за 5 минут давал понять систему.

## Самопроверка (чеклист)

Поставьте галочки честно — это и есть ваша «оценка».

**C4**
- [ ] На Context нет контейнеров/классов — только система, акторы и внешние системы
- [ ] Видна граница системы: однозначно понятно, что наше, а что внешнее
- [ ] Уровни не смешаны (контейнеры и компоненты на разных диаграммах)
- [ ] У каждого контейнера подписаны технология и назначение
- [ ] Есть легенда / диаграмма читается без устных пояснений

**ADR**
- [ ] У каждого ADR есть Status, Context, Decision, Consequences
- [ ] В Consequences указаны и плюсы, и минусы/цена
- [ ] Decision формулируется одним предложением, а не абзацем
- [ ] Решения — реально значимые (а не «назвали переменную так-то»)

**arc42**
- [ ] Quality goals выражены **в цифрах** (не «быстро», «надёжно»)
- [ ] §5 ссылается на C4, §9 — на ADR (документ связный, а не четыре отдельных файла)
- [ ] Заполнено только нужное ядро — нет пустых разделов «для галочки»

**Общее**
- [ ] Документацию можно открыть и за 5 минут понять систему — то, чего хотел бизнес из кейса
- [ ] Нет анти-паттернов: Wall of Diagrams (лишние диаграммы) и Documentation Theater (для вида)
